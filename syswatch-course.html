Syswatch: From /proc to Terminal Metrics
01

What Syswatch Is When It Runs

You launch one binary, and it repeatedly samples Linux kernel counters from /proc, computes rates, and prints a row every tick.

Runtime Story in 6 Beats

This tool is a narrow loop with one job: collect snapshots, compute deltas, render output. The key idea is that rates like CPU and network are delta-based, so one warm-up interval is required.

Parse flags
Read baselines
Sleep interval
Read current stats
Compute rates + percentages
Print row (human or CSV)
Aha

The first printed row is intentionally delayed whenever CPU/disk/network are enabled, because you need two snapshots to get a per-second rate.

Plain English

Start counting output rows from zero.

If any metric needs deltas, enforce an initial sleep before the first row.

Keep looping until Ctrl+C sets the stop flag, or until iterations are exhausted.

Each pointer will either point to valid fresh stats or remain NULL if that collector failed this tick.

Code ↔ English: Conditional Baseline Math

Code 
if (cfg.show_disk && disk_read_snapshot(&disk_curr) == 0) {
	if (disk_ready && disk_compute_stats(&disk_prev, &disk_curr, cfg.interval_sec, &disk_stats) == 0) {
		disk_ptr = &disk_stats;
	}
	disk_prev = disk_curr;
	disk_ready = true;
}
Plain English

Only work disk metrics when that feature is enabled and current snapshot read succeeded.

Rate computation runs only when we already have a previous snapshot (the warm-up condition).

If successful, publish this tick's disk stats to the renderer.

Always roll current into previous so the next iteration has fresh baseline data.

04

Data Model and Transformations

Counters become deltas, deltas become rates, rates become formatted strings, and finally one row appears on stdout.

Data Flow Animation

/proc snapshots
struct arrays
delta + normalize
rendered row

Each collector stores compact fixed-size arrays (MAX_DISKS, MAX_INTERFACES, MAX_PROCESSES). That keeps memory predictable and avoids per-tick heap churn.

Code ↔ English: Memory Data Derivation

Code 
if (out->mem_total >= out->mem_available) {
	out->mem_used = out->mem_total - out->mem_available;
}
if (out->swap_total >= out->swap_free) {
	out->swap_used = out->swap_total - out->swap_free;
}
Plain English

Compute used RAM from total minus available, but only if values are sane.

Use the same guardrail for swap to prevent underflow when kernel counters are unexpected.

These two derived fields are what rendering code actually prints.

Code

06

Engineering Patterns Inside the Code

The design leans on consistent patterns: parse-then-validate, snapshot-then-delta, aggregate-then-format, and graceful degradation.

Pattern Cards

Input Contracts Early

Helpers like parse_positive_int and parse_non_negative_int reject invalid CLI values before runtime loop starts.

Local Failure Containment

Collectors can fail independently. Renderer checks null pointers and keeps printing usable rows.

Fixed-Capacity Buffers

MAX_* constants provide deterministic memory usage and predictable upper bounds for per-tick work.

Single Render Gateway

All presentation logic stays in display.c, making human and CSV mode switches cheap and centralized.

Code ↔ English: Human vs CSV Header Strategy

Code 
if (cfg->csv_mode) {
	printf("timestamp");
	if (cfg->show_cpu) {
		printf(",cpu_usage_pct,cpu_user_pct,cpu_system_pct,cpu_idle_pct");
	}
	printf("\n");
	return;
}
Plain English

When CSV mode is active, renderer emits machine-friendly column names first.

CPU headers are appended only when that feature is enabled, so output schema matches selected flags.

Early return cleanly separates CSV path from colored human-table path below.

Aha

The display module is effectively a strategy switch keyed by one boolean: csv_mode. Everything else reuses shared stats structures.

07

Edge Cases and Failure Modes

This module highlights where data can be missing or unstable, and how the code recovers without hard failure.

What Can Go Wrong, and Where

Process Exit Race

/proc PID may disappear between directory scan and stat read; collector simply skips that process.

Counter Resets

If current counters are less than previous, deltas are forced to zero to avoid negative rates.

Division by Zero

When interval or total jiffy delta is zero, code guards and emits zero instead of crashing.

Failure Containment

No collector failure path aborts the monitor loop. The design favors continuity over completeness for any single row.

Code ↔ English: Defensive Disk Delta

Code 
read_delta = (c->sectors_read >= p->sectors_read) ? (c->sectors_read - p->sectors_read) : 0;
write_delta = (c->sectors_written >= p->sectors_written) ? (c->sectors_written - p->sectors_written) : 0;

dst = &out->items[out->count++];
snprintf(dst->name, sizeof(dst->name), "%s", c->name);
dst->read_bps = ((double)read_delta * DISK_SECTOR_SIZE) / (double)interval_sec;
Plain English

Only trust monotonic growth; otherwise treat that metric as no-change this tick.

Write into the next output slot after both deltas are stable.

Convert sector delta into bytes/sec with fixed sector size and configured interval.

Big Picture

If You Need to Add a Metric

Create a new collector file with snapshot + compute functions, add fields to structs in syswatch.h, wire into main loop and display_print_row.

If You Need Better Throughput

Profile process_read_list first: /proc scan and file opens dominate cost. Consider caching pid metadata or lower scan frequency.
